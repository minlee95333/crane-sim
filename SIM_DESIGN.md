# 현실 기반 양중 시뮬레이터 설계 (SIM_DESIGN)

> 목적: **양중 스케줄링 최적화 RL**을 위한, 현실에 기반한 양중 시뮬레이터 설계.
> 이 문서는 "무엇을 왜 현실적으로 모델링할지"와 "기존 `crane-rl-dash-auto-reward. V2`와 어떻게 연동할지"를 정의한다.
> 구현 로드맵은 티어별 우선순위로 제시하되, 코드는 아직 착수하지 않는다(문서 우선).

---

## 0. 배경 — 두 개의 층위

양중 RL은 서로 다른 두 층위가 있고, 필요한 "현실성"이 다르다.

| 층위 | 결정 대상 | 예시 시스템 | 필요한 현실성 |
|---|---|---|---|
| **거시(스케줄링)** | 어느 크레인이 · 어디 서서 · 어떤 순서로 양중 | V2 `crane_core/env.py` (MAPPO) | 사이클 타임, 비용, 타당성, 시공순서, 다중크레인 충돌 |
| **미시(조작)** | 후크를 실제로 3D로 비행 (선회/기복/권상) | `crane-sim` `Environment.js` | 물리, 흔들림, 충돌, 모멘트 리미터 |

**핵심 방침 (2026-07-02 확정):** 최적화 대상은 **여러 양중물이 있는 계획 단계** — 거시 스케줄링이다. 양중물 1건의 조작이 아니라 "N개의 양중물을 M대의 크레인으로 어떤 배치·순서로 처리하는가"를 시뮬레이션하고 최적화한다. 현재 V2는 사이클 타임·비용을 **규칙 기반 추정치**로 쓴다. 현실 시뮬레이터의 진짜 가치는 이 추정치를 **물리·현장 제약으로 검증된 값**으로 대체하는 것. 따라서 (1) crane-sim에 **계획 계층**(양중 단위 실행·타임라인)을 얹고, (2) 추가할 현실 요소는 "**스케줄의 타당성·시간·비용을 바꾸는 것**"을 우선한다. 조이스틱 수준 미시 조작은 최종 목표가 아니라 계획 계층의 **실행 엔진**이다.

```
[거시 스케줄러 RL]  ── 어느 크레인/셋업/순서 ──┐
        ▲                                      │  "이 lift, 이 셋업에서 가능? 사이클 몇 분? 비용은?"
        │  실제 사이클타임·타당성·비용            ▼
        └────────── [현실 양중 시뮬레이터 (crane-sim 확장)] ──── ground truth
                     정격·안정성·간섭·리깅타임·바람 → 타당성 + 소요시간/비용
```

## 1. 현재 자산 (이미 있는 것)

### crane-sim (미시)
- 계층 구조: `core`(World/Crane/Load/LoadChart) → `sim`(Simulation, Environment, Recorder) → `render` → `control`
- 기구학: 이동식(선회/기복/권상), 타워(트롤리), 고정스텝 결정론
- 정격: **1D 반경→하중** 로드차트 + 모멘트 리미터(과하중 시 악화동작 차단)
- 픽업/안착(목표 판정), 장애물 AABB 충돌, 인양 금지구역, 후크 흔들림(옵션 펜듈럼)
- RL 인터페이스: `reset/step(action)→{obs,reward,done,info}`, 시나리오 7종, 기록/리플레이

### V2 (거시) — 이미 모델링된 현실 요소
- **2D 정격표**: (붐길이 × 반경) 차트, `delta_z`(높이차) 반영, 무게별 최대반경
- **제한구역**: 사각형, 경로 차단 판정, 우회 최단경로(코너 웨이포인트 + clearance)
- **셋업**: 셋업 위치 선택(가능 디스크), setup/travel/teardown/duration 시간축
- **비용**: 임대·유휴·연료·노무 (`cost.py`, KRW 단가 튜너블)
- **점수**: 완료·makespan·soft간섭·부하균형 (`scorer.py`)
- **다중 크레인**: MAPPO, hard/soft 간섭, 커리큘럼
- **IRL**: 사람 플레이 → Conditional MaxEnt IRL prior, 철골 부재 생성(`steelframe`)

### 두 프로젝트의 공백 (둘 다 없는 것)
바람, 아웃리거/지반 지지력, 전도 안정성, 테일 스윙, 리깅 사이클 타임, 시공순서(precedence), 부재 반입 스케줄, 무게중심/슬링 기하.

## 2. 추가할 현실 요소 (티어별)

### 🔴 Tier 1 — 스케줄 결과를 실제로 바꾸는 것 (필수)

#### T1-① 양중 사이클 타임 (리깅 + 미세 정렬)
- **문제**: 현재 crane-sim은 Space 즉시 픽업/해제. 현실 사이클의 30~50%가 줄걸이·신호·시험인양·태그라인 정렬.
- **모델**:
  - `rigTime`, `derigTime` — 부재 무게·형상(길이/수풍면적)·인양점 수에 따른 준비 시간
  - **미세 정렬 페널티**: 안착 마지막 1~2 m는 속도 급감(정밀 위치잡기). 후크 흔들림이 클수록 더 느려짐(흔들림 안정 대기)
  - `trialLift` — 첫 인양 시 지면 이격 후 정지·확인 시간
- **영향**: makespan·비용 현실화. **스케줄링 RL에 가장 직결.**
- **효과 대비 난이도**: 높음 / 낮음. World의 attach/release를 즉시 토글 → 시간 소요 상태기계로.

#### T1-② 2D/3D 정격표
- **문제**: crane-sim은 반경 1D. 실제 정격은 (붐길이 × 반경), 방향(오버프론트/오버사이드), 지브 구성에 따라 다름.
- **모델**:
  - (붐길이 × 반경) 2D 차트 — V2 `_normalize_capacity_chart` 이식
  - **오버프론트/오버사이드/오버리어** 구분(아웃리거 방향별 정격 차등)
  - 붐 텔레스코핑(길이 가변), 지브/러핑지브, **후크블록·리깅 공제(net capacity)**
  - **동하중계수**(충격) 1.1~1.25 — 인양은 정적이 아님
- **영향**: "이 구성에서 이 무게 가능?"의 정확도 → 타당성 판정.
- **난이도**: 중 / 중. LoadChart를 2D 보간 + 방향 인자로 확장.

#### T1-③ 크레인 안정성 (전도 / 지반)
- **문제**: 정격표는 "구조 강도"만. 실제로는 **전도(tipping)**·지반 지지력이 별도 한계.
- **모델**:
  - 아웃리거 반력 계산(하중·반경·카운터웨이트·붐 자중 모멘트) → 각 아웃리거 접지압
  - 지반 지지력(scenario의 지반 등급) vs 접지압 → **셋업 가능 여부**
  - 아웃리거 확장폭/매트(pad) 크기 옵션
- **영향**: **셋업 위치 선택**을 물리로 제약 → 스케줄링의 핵심 결정 변수.
- **난이도**: 중 / 중. 준정적 모멘트 균형으로 충분(전산 유체 불필요).

#### T1-④ 테일 스윙 & 크레인 간 간섭
- **문제**: crane-sim 다중 크레인은 있으나 서로 물리 간섭 없음.
- **모델**:
  - **테일 스윙**: 카운터웨이트 후미 회전 반경 → 인접 크레인·구조물·경계와 충돌
  - **붐/지브 교차**: 두 크레인 붐의 3D 근접 판정
  - **작업반경 겹침** → 동시작업 금지 구역·시퀀싱 제약
- **영향**: 다중 크레인 배치·순서를 결정. V2의 soft/hard 간섭을 **물리 근거로 승격**.
- **난이도**: 중 / 중. 회전 스윕 영역 + 선분 근접 판정.

### 🟡 Tier 2 — 현실감·타당성 크게 상승

#### T2-⑤ 부재 반입·야적(laydown) & 반출
- 부재가 시간에 따라 도착(delivery schedule), 야적장 버퍼 관리, 반출 → "언제 무엇을 들 수 있나"의 시간 창.

#### T2-⑥ 시공 순서 제약 (erection precedence)
- 기둥 → 보 → 슬래브. 선행 부재 없이 후속 배치 불가 → 스케줄러가 지켜야 할 **DAG**.
- V2 `steelframe`에 부재 생성 있음 → 부재에 `dependsOn` 필드·layer 순서 부여.

#### T2-⑦ 바람
- 최대 작업 풍속(부재 수풍면적별), 풍속→작업중단/흔들림 증가 → **작업 가능 시간대(weather window)**.
- 스케줄 실현성에 직결. Sway에 바람 외력 항 추가로 물리 연결 가능.
- **물리 연동 ✅ (P7.9)**: wind에 `dir`·`gust`(결정론 시간 함수), World가 풍속²×수풍면적/질량으로
  크레인별 외력 가속을 주입 — sway 켠 크레인만 소비, 기존 timeline 정의는 동작 불변.

#### T2-⑧ 픽앤캐리(주행 인양) 감격 & 셋업/이동/조립 시간 ✅ (P7.7)
- 크롤러가 하중 걸고 주행 시 정격 급감, 크레인 이동·조립/해체 시간(대형 크레인은 수 시간~일).
- V2에 setup/travel/teardown 시간축 있음 → 물리 근거 보강.
- **구현**: 감격 정격 = 정적 × `rating.pickCarryFactor`(0.66). 주행 중 전도는 정적 모멘트 +
  종방향 가속 하중전이 동적 항. 잘 실행된 캐리(하중을 몸체 가까이·낮게)에서는 감격 정격이
  실질 제약이고, 전도는 과중·과속·큰 캐리반경에서 문제. AutoPilot 캐리 페이즈가 베이스를
  주행시켜 한 셋업으로 픽업·목표 동시 도달 불가한 양중을 완주(S10).

### 🟢 Tier 3 — 고급(조작 숙련 / IRL / 정밀도)

- **T3-⑨ 무게중심·슬링 기하**: 다점 인양 슬링 각도→장력, 편심 하중, 부재 회전 자세 요구(현재 위치만).
  — 부재 요(yaw) 회전 상태는 ✅ (P7.9, `physics.loadYaw` — 선회 지연 추종·외접 AABB 충돌 정합).
  targetYaw(안착 자세 요구)·슬링 장력은 미구현.
- **T3-⑩ 로프 동역학 심화**: 이중진자, 감쇠, 태그라인 제어(기본 Sway 있음). 조작 난이도·IRL 보상용.
  — 이중진자(후크→부재 2단) ✅ (P7.9, `physics.doublePendulum` — 기본 OFF, S11만 ON).
- **T3-⑪ 환경 상세**: 작업반경 내 기존 구조물/전력선/고도 제한, 야간/교대/일조.
  — **지상 인원·장비 동적 간섭 ✅ (P7.10)**: 시드 결정론 배회 인원·순환 차량,
  매달린 하중 위험반경 진입 시 작업 홀드(신호수 규칙) → 실측 사이클 타임 증가.
  `scenario.agents` 정의된 시나리오(S12)만 활성 — 계획 추정(근사 모드)은 이 지연을
  모르므로 "규칙 vs 물리" 차등의 새 원천. 전력선·야간 등은 미구현.

## 2.5 계획 계층 아키텍처 (신규 — 이 프로젝트의 중심)

여러 양중물의 계획 최적화를 위해 crane-sim 위에 3개 컴포넌트를 얹는다.
기존 `Environment.js`(조이스틱 RL)는 유지하되 부차 트랙으로 격하한다.

```
┌──────────────────────────────────────────────────────────┐
│ PlanEnvironment  (P: 계획 수준 RL 인터페이스 — 맨 마지막)      │
│   행동: 다음 (크레인, 양중물[, 셋업 위치]) 선택                │
│   관측: 미완료 양중물·크레인 상태·타임라인                     │
│   보상: -makespan, -비용, -안전위반 (V2 scorer 가중치 재사용)  │
├──────────────────────────────────────────────────────────┤
│ PlanRunner  (계획 실행기)                                   │
│   입력: LiftPlan = [(craneId, loadId, setupPos?), ...] 순서  │
│   실행: 크레인별 병렬 진행 + 간섭 시 대기/양보 규칙             │
│   출력: 타임라인(이벤트 로그), makespan, 비용, 안전 카운트      │
├──────────────────────────────────────────────────────────┤
│ AutoPilot  (양중 1건 자동 조종 — 실행 엔진)                   │
│   입력: (크레인, 양중물, 목표)                                │
│   행동: 권하→줄걸이→권상→선회/기복→미세정렬→안착→해체를          │
│         물리 시뮬 위에서 자동 완주 (envtest 스크립트 정책 일반화) │
│   출력: cycleTime, 경로, 안전 이벤트 — 결정론                  │
├──────────────────────────────────────────────────────────┤
│ 기존: World/Crane 물리 코어 (고정스텝 결정론)                  │
└──────────────────────────────────────────────────────────┘
```

- **AutoPilot이 곧 "현실 기반 사이클 타임 오라클"**: V2의 규칙 기반 duration 추정을 대체하는 값이 여기서 나온다. Tier 1 현실 요소(리깅 타임·미세정렬·정격·간섭)는 AutoPilot 실행에 직접 반영된다.
- **속도 전략**: 계획 탐색은 양중 1건당 수천 물리 스텝이 비싸므로, AutoPilot은 (a) 전체 물리 실행(정밀) 외에 (b) **분석적 근사 모드**(선회각/기복량/권상량을 속도·가속 램프로 적분한 닫힌식 + 리깅 타임)를 제공한다. 근사 모드는 물리 모드로 주기 검증(칼리브레이션)한다.
- **행동 공간(계획 RL)은 V2와 동형**: (크레인 × 양중물) 선택 — V2 MAPPO의 candidate 구조와 호환되어 나중에 V2 학습 코드·IRL prior를 재사용할 수 있다.

## 3. 데이터 모델 확장 (스케치)

```js
// Crane 확장
{
  ...기존,
  capacityChart: [[boomLen, [[radius, cap], ...]], ...], // 2D (T1-②)
  outrigger: { spanX, spanY, padArea },                  // (T1-③)
  counterweight: { mass, radius },                       // 자중 모멘트·테일스윙
  tailSwingRadius, boomFootprint,                        // (T1-④)
}

// Load 확장
{
  ...기존,
  cog: [x, y, z],           // 무게중심 (T3-⑨)
  liftPoints: [[...], ...], // 인양점 → 슬링 각도
  windArea,                 // 수풍면적 (T2-⑦)
  dependsOn: [loadId, ...], // 시공순서 (T2-⑥)
  arriveTime, laydownPos,   // 반입 스케줄 (T2-⑤)
  targetYaw,                // 안착 자세 요구 (T3-⑨)
  rigTime, derigTime,       // (T1-①)
}

// Scenario / World 확장
{
  ground: { bearingCapacity, grade },   // (T1-③)
  wind: { speed, dir, gust, maxOperating }, // (T2-⑦)
  deliveries: [{ loadId, time }, ...],  // (T2-⑤)
  agents: {                             // (T3-⑪, P7.10) 지상 인원·장비 — 시드 결정론
    seed, dangerRadius,                 //   같은 시드 = 같은 궤적 (리플레이·RL 재현)
    workers: [{ count, area, speed, idle }],
    vehicles: [{ route, speed, size }],
  },
}
```

## 4. V2 연동 방안

목표: V2 스케줄러가 규칙 기반 추정 대신 **crane-sim을 ground-truth 오라클**로 호출.

1. **feasibility/cycle-time 오라클 API**
   `evaluateLift(crane, setup, load, siteState) → { feasible, cycleTime, cost, violations }`
   - crane-sim이 헤드리스로 해당 lift 1건을 시뮬레이션(또는 준정적 계산)해 반환.
2. **교체 지점**: V2 `crane_core/env.py`의 `candidate_outcome`(현재 setup/travel/duration 추정) → 오라클 호출로 대체 가능한 인터페이스로.
3. **비용/점수 재사용**: V2 `cost.py`/`scorer.py`의 단가·가중치는 그대로. 입력이 되는 시간/타당성만 시뮬레이터 산출값으로.
4. **언어 경계**: crane-sim은 JS, V2는 Python. 옵션 — (a) 오라클을 Python으로 포팅, (b) crane-sim 헤드리스를 Node 서브프로세스/HTTP로 호출, (c) 핵심 계산(정격·안정성·간섭)만 공용 스펙(JSON)으로 양쪽 구현. **권장: (c)** — 정격표·제약을 단일 데이터 소스로 두고 양쪽이 읽음.

*P1~P2가 최우선: 이 둘이 되면 "여러 양중물의 계획을 넣으면 현실적 타임라인이 나오는" 시뮬레이터가 완성되고, 이후 현실 요소(P3~P6)는 그 정확도를 올리는 작업이 된다. 베이스라인 계획(greedy/최근접)과의 비교도 P2에서 바로 가능.*
